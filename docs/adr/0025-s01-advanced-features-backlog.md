## 背景

D19 用户真机手验,反馈 S01 手动记账偷懒 3 处高级功能:

| # | 功能 | 偷懒原因 | 影响 |
|---|---|---|---|
| 1 | **撤销确认**(误删恢复)| MVP 简化 | 用户误删交易 → 数据丢失,无撤销 |
| 2 | **重复检测**(同金额同日提示)| MVP 简化 | 用户漏记 / 重复记账无法自动提示 |
| 3 | **汇率换算**(出国记账)| MVP 简化 | 出国消费只能手动换算,易错 |

**根因**:S01 阶段只关注「5 秒 3 步记账」核心路径,没考虑「边界场景」(误删、重复、出国)。

---

## 决策

**v1.0 不做,ADR 记录 v1.1 评估。**

WHY:
- S01 阶段「5 秒 3 步」核心已达成(CLAUDE.md §1 「极致体验」)
- 3 个偷懒都是**进阶功能**,不是核心
- v1.1 时间预算(4-6 周)允许补这些

## v1.1 实施计划

### 1. 撤销确认(优先级:高)

**用户场景**:误删交易 → 弹 SnackBar「已删除,撤销」,5 秒内可点「撤销」恢复

**技术方案**:
- `transactions` 表加 `deleted_at INTEGER NULL` 列(软删除,schema v6)
- 删除改为 `UPDATE transactions SET deleted_at = ?`(不是 `DELETE`)
- 主页查询 `WHERE deleted_at IS NULL`
- SnackBar「撤销」按钮调 `UPDATE transactions SET deleted_at = NULL WHERE id = ?`

**WHY 软删除 vs 真删除**:
- 用户体验:误删可恢复
- 数据完整性:保持外键关系
- 实现成本:不高(SQL 改 1 行,UI 加 SnackBar action)

### 2. 重复检测(优先级:中)

**用户场景**:今天已经记了一笔「餐饮 ¥50」,用户又记「餐饮 ¥50」 → 弹「你今天记过类似交易,是否继续?」

**技术方案**:
- 提交前查 `SELECT * FROM transactions WHERE account_id = ? AND amount_cents = ? AND date(occurred_at) = date('now')`
- 命中 → 弹 AlertDialog 让用户选「继续 / 查看已记 / 取消」

**WHY 比自动去重好**:
- 自动去重可能误删用户故意的多笔同金额
- 提示 + 让用户选,符合 CLAUDE.md §1 「极致体验」(不歧义)

### 3. 汇率换算(优先级:低)

**用户场景**:出国旅游 ¥100 美金 → 记账选「餐饮 ¥100」 → 自动按当前汇率换算成 ¥700

**技术方案**:
- 选分类时多 1 步「币种选择」(默认人民币)
- 选外币时,弹「当前汇率:1 USD = 7.0 CNY」确认
- `transactions` 表加 `original_currency TEXT` + `original_amount_cents INTEGER` 列(schema v7)
- 主页显示原币 + 换算后人民币(2 行)

**WHY 优先级低**:
- 用户基数小(出国记账是少数场景)
- 汇率 API 维护成本(免费 API 不稳定)
- v1.1 可考虑简化版(手动输入汇率,不做实时 API)

## 不可逆性

| 项 | 不可变更 | 理由 |
|---|---|---|
| 软删除(不在 transactions 表加 deleted_at)| 永不变更 | 外键依赖 |
| 重复检测 = 提示而非自动去重 | 永不变更 | 用户体验决策 |
