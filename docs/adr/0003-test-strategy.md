# ADR-0003: 测试策略（70/20/10 金字塔 + 关键模块 100%）

> 状态：**已接受**
> 日期：2026-07-14
> 决策者：用户（Owner）+ Claude（执行）
> 影响范围：v1.0 全期所有代码

---

## 背景

需要为 v1.0 MVP 建立测试策略。约束：

- 涉及**金额计算**（容错为 0）
- 涉及**信用卡还款**（错一次就是逾期）
- 涉及**加密**（Stage 6）
- 长期演进（v1.0 → v2.0）
- 单人 + AI 协作

测试需要：
- 找到 BUG 不是目的，是手段
- 关键路径零 BUG
- 重构不怕
- AI 不会假装测过

---

## 决策

**采用测试金字塔 + 关键模块 100% 覆盖**

### 测试比例

```
   /\
  /  \      集成测试 10%（关键用户旅程）
 /----\
/      \    Widget 测试 20%（复杂 UI 组件）
-------- 
单元测试 70%（业务规则、计算、数据）
```

### 覆盖率目标

| 模块 | 目标 |
|---|---|
| `lib/domain/`（业务规则） | **100%** |
| `lib/data/repositories/` | **≥ 90%** |
| `lib/data/services/` | **≥ 90%** |
| `lib/features/*/domain/` | **≥ 90%** |
| `lib/features/*/data/` | **≥ 80%** |
| `lib/presentation/` | **≥ 60%** |
| **整体** | **≥ 80%** |

### 关键模块 100% 覆盖

- 金额计算（任何 0.01 误差都不行）
- 信用卡还款
- 预算执行
- 数据加密（Stage 6）
- 攒攒台词触发（Stage 7）

---

## 理由

1. **金额类业务必须 100%**：1 分钱错就是用户损失
2. **关键模块 100%**：这些地方 BUG 代价巨大
3. **UI 测试适度**：UI 变化快，过度测试成本高
4. **TDD 默认**：复杂逻辑先测试，避免 AI 假装通过
5. **覆盖率可量化**：CI 检查，达标才能 OWNER_ACCEPTANCE

---

## 替代方案

### 方案 A：无测试，靠 AI "感觉"

- ❌ 完全不靠谱，AI 不会报错
- ❌ 用户不能验收

**为什么没选**：等于裸奔

---

### 方案 B：100% 覆盖率

- ✅ 覆盖最完整
- ❌ UI 测试成本极高
- ❌ 时间不允许（8 周太紧）

**为什么没选**：性价比低

---

### 方案 C：只测关键路径（关键 100%，其他 0）

- ❌ 集成 BUG 漏检
- ✅ 快速

**为什么没选**：漏掉中间层 BUG

---

## 后果

### 正面

- 关键模块零 BUG
- 重构有保护
- AI 不能假装通过（测试不撒谎）
- 用户验收有量化标准

### 负面

- 测试代码量 ≥ 业务代码量
- 初期开发慢 30-50%
- 需要纪律

### 风险

- 测试本身写错 → 通过但不正确
  - 缓解：边界条件 + 错误路径覆盖
  - 缓解：code review
- 时间不够 → 跳过测试
  - 缓解：覆盖率不达标不能 ROA
- AI 写假测试 → 用例不实际验证
  - 缓解：禁止"空测试"（test 占位不算）

---

## 实施细节

### TDD 流程（关键模块）

```
1. Red:    写失败的测试
2. Green:  写最小实现
3. Refactor: 重构
4. Repeat
```

### 测试纪律

- ✅ 改代码必改测试
- ✅ 测试不通过不能 commit
- ✅ 边界条件必须覆盖（0、null、空、最大、最小）
- ✅ 错误路径必须覆盖
- ❌ 不写"空测试"（`test('TODO', () {});`）
- ❌ 不测实现细节（测行为）

### CI 集成（Stage 8）

```yaml
- name: Test with coverage
  run: |
    flutter test --coverage
    # 检查覆盖率
```

详细规范：[governance/test-strategy.md](../governance/test-strategy.md)

---

## 复盘条件

如果出现以下情况，重新评估：

1. 测试代码量超过业务代码 2 倍（可能过度）
2. 关键模块 BUG 仍然漏检（测试方法有问题）
3. v2.0 加新平台（测试策略需要调整）
4. 集成测试频繁 flake（环境问题）
