# ADR-0003: 测试策略（60/20/30/10 四层金字塔 + 关键模块 100%）

> 状态：**已接受**（v1.1 — 2026-07-21 增 E2E 第四层）
> 日期：2026-07-14 · 更新：2026-07-21（增 ADR-0014 E2E）
> 决策者：用户（Owner）+ Claude（执行）
> 影响范围：v1.0 全期所有代码

---

## 背景

需要为 v1.0 MVP 建立测试策略。约束：

- 涉及**金额计算**（容错为 0）
- 涉及**信用卡还款**（错一次就是逾期）
- 涉及**加密**（Stage 6）
- 长期演进（v1.0 → v2.0）
- 单人 + AI 协作

测试需要：

- 找到 BUG 不是目的，是手段
- 关键路径零 BUG
- 重构不怕
- AI 不会假装测过

---

## 决策

**采用四层测试金字塔 + 关键模块 100% 覆盖**

### 测试比例

```
         /\            E2E 10%（integration_test，真 Flutter engine + 真 SQLite）
        /  \           ADR-0014 — Stage 6/7 关键路径必真引擎跑
       /----\
      /  Widget \      Widget + 模块集成 30%（复杂 UI / 跨模块）
     /------------\
    /  单元测试     \   单元测试 60%（业务规则、计算、DAO）
   /----------------\
```

> 注：原 70/20/10 调整为 60/30/10（集成 + Widget 共享 30%），腾出 10% 给 E2E。

### 覆盖率目标

| 模块 | 目标 |
|---|---|
| `lib/domain/`（业务规则） | **100%** |
| `lib/data/repositories/` | **≥ 90%** |
| `lib/data/services/` | **≥ 90%** |
| `lib/features/*/domain/` | **≥ 90%** |
| `lib/features/*/data/` | **≥ 80%** |
| `lib/presentation/` | **≥ 60%** |
| **整体** | **≥ 80%** |

### 关键模块 100% 覆盖

- 金额计算（任何 0.01 误差都不行）
- 信用卡还款
- 预算执行
- 数据加密（Stage 6 — 必须 E2E 真引擎跑）
- 攒攒台词触发（Stage 7 — 必须 E2E 真推理）

### E2E 必加场景（ADR-0014）

| Stage | 必加 E2E |
|---|---|
| S01 | 记账主流程 + emoji 真渲染 + 真持久化 |
| S02 | 分类 CRUD + 多账户 |
| S03 | 信用卡还款 |
| S04 | 预算执行 |
| S05 | 净资产计算 |
| S06 | **SQLCipher 加密 + 错误密钥拒开** |
| S07 | **AI 攒攒推理 + 异常检测** |
| S08 | 全用户旅程端到端 + iOS 16/17/18 兼容 |

---

## 理由

1. **金额类业务必须 100%**：1 分钱错就是用户损失
2. **关键模块 100%**：这些地方 BUG 代价巨大
3. **UI 测试适度**：UI 变化快，过度测试成本高
4. **TDD 默认**：复杂逻辑先测试，避免 AI 假装通过
5. **覆盖率可量化**：CI 检查，达标才能 OWNER_ACCEPTANCE
6. **E2E 补盲点**（ADR-0014）：widget test 用 fake_async + 内存 DB 覆盖不到真机场景（字体 / 文件 / Plugin / 生命周期）

---

## 替代方案

### 方案 A：无测试，靠 AI "感觉"

- ❌ 完全不靠谱，AI 不会报错
- ❌ 用户不能验收

**为什么没选**：等于裸奔

---

### 方案 B：100% 覆盖率

- ✅ 覆盖最完整
- ❌ UI 测试成本极高
- ❌ 时间不允许（8 周太紧）

**为什么没选**：性价比低

---

### 方案 C：只测关键路径（关键 100%，其他 0）

- ❌ 集成 BUG 漏检
- ✅ 快速

**为什么没选**：漏掉中间层 BUG

---

### 方案 D：只做 widget test，不做 E2E

- ❌ 真机问题堆积到 Stage 6/7/8（SQLCipher 真加密失败 / AI 推理性能未知 / 字体回退等）
- ✅ 快

**为什么没选**：用户决策（2026-07-21）"不能把问题积压到后面"

---

## 后果

### 正面

- 关键模块零 BUG
- 重构有保护
- AI 不能假装通过（测试不撒谎）
- 用户验收有量化标准
- **Stage 6/7 真机风险前置解决**（E2E 真引擎跑）

### 负面

- 测试代码量 ≥ 业务代码量
- 初期开发慢 30-50%
- 需要纪律
- **每个 Stage 写 E2E 占用 ~2-3h，Stage 8 总投入 ~16-24h**

### 风险

- 测试本身写错 → 通过但不正确
  - 缓解：边界条件 + 错误路径覆盖
  - 缓解：code review
- 时间不够 → 跳过测试
  - 缓解：覆盖率不达标不能 ROA
- AI 写假测试 → 用例不实际验证
  - 缓解：禁止"空测试"（test 占位不算）
- E2E Flaky（impeller 渲染 / 真机动画）
  - 缓解：`pumpAndSettle` + retry + 真机截图比对
- E2E 跑需要 macOS / 真机
  - 缓解：GitHub Actions 免费 macOS runner + 用户真机手跑

## 实施细节

### TDD 流程（关键模块）

```
1. Red:    写失败的测试
2. Green:  写最小实现
3. Refactor: 重构
4. Repeat
```

### 测试纪律

- ✅ 改代码必改测试
- ✅ 测试不通过不能 commit
- ✅ 边界条件必须覆盖（0、null、空、最大、最小）
- ✅ 错误路径必须覆盖
- ✅ **每 Stage 至少 1 个 E2E**(ADR-0014)
- ✅ Widget 加 Key 便于 E2E 查找(`Key('语义')`,Day 8 强制)
- ❌ 不写"空测试"（`test('TODO', () {});`）
- ❌ 不测实现细节（测行为）

### CI 集成（Stage 8）

```yaml
- name: Unit + Widget Test
  run: flutter test

- name: E2E (iOS Simulator)
  uses: appleboy/flutter-action@v2
- run: |
    xcrun simctl boot "iPhone 15"
    flutter test integration_test/ -d "iPhone 15"
```

详细规范：[governance/test-strategy.md](../governance/test-strategy.md)、[ADR-0014](../adr/0014-e2e-testing-strategy.md)

## 复盘条件

如果出现以下情况，重新评估：

1. 测试代码量超过业务代码 2 倍（可能过度）
2. 关键模块 BUG 仍然漏检（测试方法有问题）
3. v2.0 加新平台（测试策略需要调整）
4. 集成测试频繁 flake（环境问题）
5. E2E 跑时间 > 10 分钟 / Stage（优化目标或减少 E2E 数）
