# Subagent 提示库

> 状态：`GOVERNANCE`
> 创建：2026-07-14
> 适用范围：大副调用 subagent 时使用的标准化 prompt

---

## 🎯 目的

- 复用高质量 prompt，避免每次重新设计
- 确保 subagent 输出格式一致
- 提高 subagent 调用效率

---

## 📚 Subagent 类型速查

| Subagent | 用途 | 调用时机 |
|---|---|---|
| `general-purpose` | 通用任务（执行 + 读 + 写） | 偶尔用 |
| `Explore` | 只读搜索 / 查文档 | 频繁 |
| `Plan` | 出设计方案 | 复杂决策前 |
| `statusline-setup` | 配置状态栏 | 一次性 |

**重要**：subagent 不能修改项目代码（除 statusline-setup）

---

## 🔍 1. Explore：查文档 / 搜代码

### 模板

```
请帮我查找关于 "{关键词}" 的信息。

需求：
1. [具体需求 1]
2. [具体需求 2]

输出要求：
- 文件路径（如适用）：[file:line]
- 文档链接（如适用）：[URL]
- 关键代码片段（如适用）：[code]
- 不要修改任何文件

完成后返回 [结果摘要]，不要做其他事。
```

### 示例：查 SQLCipher 用法

```
请帮我查找关于 "Flutter SQLCipher 集成" 的信息。

需求：
1. 最新版本号和 pubspec.yaml 配置
2. iOS 平台配置步骤
3. 密钥管理最佳实践
4. 已知问题和解决方案

输出要求：
- 文档链接：[URL]
- 关键代码片段：[code]
- 不要修改任何文件

完成后返回搜索摘要，不要做其他事。
```

## 🏛️ 2. Plan：出设计方案

### 模板

```
请帮我设计 "{功能/问题}" 的实现方案。

背景：
- 项目：iOS 自用记账 App《审计官》
- 技术栈：Flutter 3.24+ / Riverpod / Drift
- 当前 Stage：[N]
- 相关文件：[paths]

约束：
- [约束 1，如"不能引入新依赖"]
- [约束 2，如"性能预算"]
- [约束 3，如"兼容 iOS 16+"]

请输出：
1. **方案概述**：1-2 段话
2. **架构图**：用 ASCII 或描述
3. **文件改动列表**：具体到文件路径
4. **依赖变更**：是否需要新 package
5. **风险评估**：🟢/🟡/🔴
6. **回滚方案**：如何撤销
7. **测试策略**：怎么验证

不要写代码，只出方案。完成后返回方案概要，不要做其他事。
```

### 示例：Stage 6 加密方案

```
请帮我设计 "SQLCipher + Keychain 集成方案"。

背景：
- 项目：iOS 自用记账 App《审计官》
- 技术栈：Flutter 3.24+ / Riverpod / Drift 2.20+
- 当前 Stage：S06 存储 & 快照

约束：
- 密钥必须存 iOS Keychain
- 性能：启动 + 加密开销 ≤ 200ms
- 不引入额外依赖（如 sqflite_sqlcipher）
- 兼容现有 Drift schema
- 需要支持数据快照（加密导出）

[详细 prompt 略]

不要写代码，只出方案。
```

## 🔍 3. 督察（Auditor）：代码审计

### 模板

```
你是督察 🔍⚖️，只读审计员。请审计 Stage {N} 的代码。

审计范围：
- 代码文件：[lib/feature/xxx/]
- Commit 范围：[SHA1..SHA2]
- 文档：[docs/stages/S{N}-*.md]

审计维度（按重要性）：
1. **代码与 PLAN 一致性**：实现是否完全按 PLAN
2. **测试覆盖率**：≥ 80%（关键模块 100%）
3. **lint / 类型检查**：flutter analyze 必须 0 issues
4. **安全隐患**：SQL 注入 / 密钥泄露 / 信息泄露
5. **性能问题**：N+1 查询 / 不必要的 rebuild / 内存泄漏
6. **文档完整性**：daily / ADR 是否更新
7. **Git 历史**：commit 是否规范

输出格式（严格按此）：

## 总览
| 维度 | 结果 |
|---|---|
| **总体** | ✅ PASS / ⚠️ PASS WITH NOTES / ❌ FAIL |
| ... |

## 🔴 高危发现（必须修复）
### F-{NN}: [标题]
- 位置：[file:line]
- 类别：代码 / 安全 / 性能 / 测试 / 文档
- 描述：[详细]
- 证据：[命令输出 / 代码片段]
- 建议：[修复]
- 阻塞验收：✅ 是

## 🟡 中等发现（建议改进）
### F-{NN}: [标题]
- 位置：[file:line]
- ...（同上）

## ✅ 做得好的

## 审计结论
✅ PASS / ⚠️ PASS WITH NOTES / ❌ FAIL

## 签字
- 审计员：督察 subagent
- 日期：[YYYY-MM-DD]

⚠️ **不要修改任何文件**，只读。完成后返回审计报告，不要做其他事。
```

## 🍎 4. 苹果匠：Swift / iOS API 研究

### 模板

```
请研究 "{Apple API}" 的最新用法。

需求：
1. API 用途和适用场景
2. Swift 实现代码示例（5.x 语法）
3. Flutter 集成方式（Pigeon 桥接优先）
4. Info.plist 需要添加的 key
5. 测试方法和模拟器限制
6. 已知问题和官方文档链接

输出要求：
- 代码示例要完整可运行
- 引用 Apple 官方文档（developer.apple.com）
- 不要修改任何项目文件

完成后返回研究结果，不要做其他事。
```

### 示例：Vision API OCR

```
请研究 "iOS Vision API 文字识别（OCR）"。

需求：
1. VNRecognizeTextRequest 用法
2. 识别中文 + 英文混合
3. 准确度选项
4. 异步处理（不阻塞 UI）
5. Flutter 集成（用 Pigeon）

输出要求：
- 完整 Swift 示例代码
- Pigeon schema 定义
- Info.plist 配置（无需相机权限，但要相册）
- 不要修改任何项目文件

完成后返回研究结果。
```

## 🤖 5. 攒攒师：Prompt 设计

### 模板

```
请设计 "{任务}" 的 LLM prompt。

任务背景：
- 用途：[说明，如"识别交易异常"]
- 输入：[输入格式，如"交易列表 JSON"]
- 输出：[期望输出格式]
- 模型：[用哪个模型]

约束：
- 中文输出
- 输出可被代码解析（JSON 优先）
- 不输出敏感信息
- Prompt 可调（参数化）

输出要求：
1. **Prompt 草稿**（完整文本）
2. **Few-shot 示例**（3-5 个输入输出对）
3. **预期输出格式**（JSON Schema）
4. **失败兜底**（LLM 不可用时的 fallback）
5. **成本估算**（每次调用 token 数）

不要修改任何文件。完成后返回 prompt 草稿。
```

## 📐 6. 通用：复杂任务派单

### 模板

```
你是 [角色名]，请完成 [任务]。

任务背景：
- 项目：iOS 自用记账 App《审计官》
- 当前 Stage：[N]
- 相关文件：[paths]
- 相关 ADR：[adr-XXXX]

任务说明：
[详细任务描述]

输入：
- [输入 1]
- [输入 2]

输出要求：
1. [输出 1 - 明确格式]
2. [输出 2]
3. [输出 3]

约束：
- [约束 1]
- [约束 2]

完成后返回 [输出概要]，不要做其他事。

⚠️ 重要：
- 不要修改项目代码
- 不要修改 SSOT 文件（PLAN/ROADMAP/CONTROL_TOWER）
- 如有不确定，写在"需要大副确认"部分
```

## 🚨 7. 紧急：安全审计

```
你是督察 🔍⚖️，请紧急审计 "{可疑代码/改动}"。

怀疑问题：
- [类型：如"密钥泄露" / "SQL 注入" / "XSS"]

审计范围：
- Commit：[SHA]
- 文件：[paths]

请：
1. 立即读取相关文件
2. 检查问题是否存在
3. 评估影响范围
4. 输出修复建议

⚠️ 紧急模式：30 分钟内返回报告。
```

## 📌 调用约定

### 大副调用 subagent 前

```
1. 明确任务目标
2. 选择合适的 subagent 类型
3. 用对应模板填充 prompt
4. 设置超时（默认 5 分钟）
5. 不要让 subagent 修改项目代码（除 statusline-setup）
```

### subagent 返回后

```
1. 大副验证输出合理性
2. 整合进主对话上下文
3. 决定是否需要再次派单
4. 通知用户
```

---

## 🔧 提示优化技巧

1. **明确角色**：subagent 表现更好
2. **给上下文**：背景 + 约束 + 相关文件
3. **指定输出格式**：用模板减少歧义
4. **禁止修改文件**：明确说，避免误改
5. **明确终止条件**："完成后返回 X，不要做 Y"

---

## 📌 Subagent 不能做的事

- ❌ 修改项目代码（lib/, test/）
- ❌ 修改 SSOT 文件（PLAN/ROADMAP/CONTROL_TOWER）
- ❌ 跑 git push
- ❌ 跑 flutter build / flutter run
- ❌ 跑需要用户输入的命令
