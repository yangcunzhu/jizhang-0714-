🎯 SSOT 思维
所有决策有单一真源，其他文档引用而不复制。冲突时查真源，派生文档自动算。
核心
📦 Stage 边界
把"无限开放"的开发切成可审计的小块（≤ 1 周）。每个 Stage 有write-set 授权包络。
核心
🔍 检查池
每个 Stage 收尾必跑 4 件套审计（lint / test / analyze / audit），不通过 = 不能进下一 Stage。
质量
📐 模板化
所有重复性产出物模板化（Stage/ADR/Task/Daily/Commit/Audit）。先建模板，再填内容。
效率
🎭 角色化
把 AI 的多面能力拆成可标识角色（指挥官/大副/架构师/苹果匠/攒攒师/督察），每个角色有专属规范和话术。
协作
🛡 4 层防御
抗幻觉 = SSOT + 记忆 + 检查池 + Stage 边界。缺任何一层都会失守。
底线
反直觉：项目越大，越要花时间在"准备"上（治理 / 模板 / Stage 规划）。准备阶段的产出决定了后续 8 周的开发速度和质量。
2 文档体系 · 7 层
不是文档越多越好，而是分层清晰、互相引用
L0 · 宪法层 （AI 入口） ├── CLAUDE.md ← 项目级 AI 操作手册（必读第一） L1 · 真源层 （冲突时以这些为准） ├── product-design.html ← 产品功能真源 ├── docs/PLAN.md ← 实施计划真源 ├── docs/ROADMAP.md ← 路线图真源 ├── docs/CONTROL_TOWER.md ← 当前状态真源（DERIVED，不手填） └── 代码 + 运行证据 ← 最高优先级 L2 · 治理层 （详细规范 · governance/） ├── roles.md ← 角色身份 + 性格 ├── scripts.md ← 标准化话术 ├── checklists.md ← Stage/Daily/Commit 清单 ├── coding-standards.md ← 代码规范 ├── project-structure.md ← 目录组织 ├── test-strategy.md ← 测试策略 + 覆盖率目标 ├── design-system.md ← 颜色/字体/间距 token ├── ux-guidelines.md ← iOS HIG + 交互规则 ├── performance-budgets.md ← 性能指标 ├── security-checklist.md ← 安全检查 ├── subagent-prompts.md ← 复用 prompt 库 ├── error-catalog.md ← 已知错误 + 解决方案 └── preview-strategy.md ← 开发期预览策略 L3 · 模板层 （复制即用 · templates/） ├── stage-template.md ├── task-template.md ├── adr-template.md ├── commit-message-template.md ├── audit-report-template.md ├── decision-request-template.md ├── branch-strategy-template.md └── daily-template.md L4 · 决策层 （不可逆 · adr/） └── NNNN-{slug}.md ← 4 位序号，接受后必须遵守 L5 · 阶段授权 （scope 控制 · stages/） └── S{N}-{slug}.md ← 每个 Stage 包含 write-set L6 · 日常记录 （可追溯 · daily/） └── YYYY-MM-DD.md ← 每天做了什么 / 没做什么 / 阻塞
关键纪律：层级越往下越具体，越往上越抽象。修改任何层级前，确认对上层的影响；冲突时永远以上层为准。

SSOT 优先级（冲突时）

  1. 产品文档（功能层面）
  2. PLAN（实施层面）
  3. ROADMAP（路线层面）
  4. CONTROL_TOWER（状态层面，派生不手填）
  5. 代码 + 运行证据（最高，最终事实）

5 长期稳定性机制 · 8 大保障
8 周 56 天 = 长周期，靠"提醒"不够，要靠结构性保障
机制解决的痛点实现方式
1. SSOT 纪律"我和你记得不一样"5 层真源优先级 + 派生不手填
2. Stage 边界"scope 慢慢扩大"write-set 授权包络 + Stage 生命周期
3. 检查池"BUG 累积到最后"每个 Stage ROA 必跑 4 件套
4. ADR 决策"技术决策反复争论"ADR 一旦接受，code 必须遵守
5. 每日日志"不知道昨天做了什么"daily/YYYY-MM-DD.md 必填
6. 上下文 4 层防御"AI 幻觉 / 上下文溢出"SSOT + 记忆 + 检查池 + Stage 边界
7. 失败即停 + 升级"AI 卡住瞎做"5/15 分钟卡住分级 + DR 流程
8. 重启策略"新会话不读历史"会话切换必读 5 文件 + 问用户

上下文 4 层防御详解

Layer 1: SSOT（文件持久化）
  ↓ 解决"上下文丢失"
Layer 2: 记忆系统（MEMORY.md）
  ↓ 解决"跨会话断片"
Layer 3: 检查池（自动审计）
  ↓ 解决"AI 假装测过"
Layer 4: Stage 边界（隔离 blast radius）
  ↓ 解决"错误蔓延"
缺任何一层都会失守：只靠 SSOT 会上下文溢出；只靠记忆会信息陈旧；只靠检查池会执行慢；只靠 Stage 会缺乏全局视角。4 层必须同时存在。

失败升级路径

卡住时长动作
≤ 5 分钟自查修复（看错误信息、查代码）
5-15 分钟查 error-catalog.md + 看 ADR
> 15 分钟停下来问用户，不准猜；写 DR（Decision Request）

7 审计闭环 · 建完立刻审
不是"做完就行"，是"做完 + 审过 + 修过"

为什么需要审计闭环

  • 人眼会漏：33 个文档中 19 处链接，AI 自己检查经常漏
  • AI 会撒谎："我觉得应该对了"≠ "实际是对的"
  • 建完没审 = 没建：不一致的文档会在 4 周后引爆

审计 4 件套

工具检查什么不通过后果
flutter analyze类型 / lint编译可能失败
flutter test测试覆盖BUG 漏检
grep / Explore subagent链接 / 命名 / 一致性文档失效
人工 review逻辑 / 用户体验功能偏差

立即修 vs 抛决策

✅ 立即修（不打扰用户）：链接失效 / 命名不合规 / 数字不一致 / 占位符泄漏 / 拼写错误
⚠️ 抛 DR（让用户决策）：架构变更 / 新增依赖 / 推翻 ADR / 范围扩大 / 用户偏好相关

📝 复盘 · 沉淀了什么

今天沉淀的方法论（10 条）

  1. SSOT 思维：单一真源，分层引用
  2. Stage 边界：write-set 授权包络，scope 可控
  3. 检查池：4 件套审计，不通过不能进下 Stage
  4. 模板化：先建模板再填内容，效率 ×3
  5. 角色化：把 AI 多面性拆成可识别角色
  6. 4 层防御：SSOT + 记忆 + 检查池 + Stage 边界
  7. ADR 决策记录：重大决策不可逆，新决策替换旧决策
  8. 项目级 CLAUDE.md：宪法层入口，新会话第一必读
  9. 冲突优先级：项目级 > 全局，明确豁免/覆盖
  10. 审计闭环：建完立刻审，立即修 vs 抛 DR

复用价值

  • 下次新项目可节省 60-70% 准备时间（治理 / 模板 / Stage 拆分）
  • 质量稳定性提升（4 件套审计不会忘）
  • AI 不再"忘记"约定（CLAUDE.md + 必读清单）
  • 用户决策负担降低（DR 流程让 AI 卡住时不猜）

本方法论的局限

适用：单人 + AI 协作 / 中长期项目（4-12 周）/ 有明确真源的项目

不太适用：纯一次性脚本 / 2 人以下不需要规范的小项目 / 完全无真源（探索性实验）
