01产品愿景与定位

用户画像（虽然是自用，但要把"自己"当成一个真实用户来定义）

维度画像
年龄 / 角色30+ 职场人，有稳定收入，有 1-2 张信用卡在使用
记账痛点坚持不下来；手动记账麻烦；信用卡还款日经常忘；月底不知道钱花哪了
对工具的态度愿意尝试新工具，但要求"装上去就能用"，学习成本高立刻弃
审美偏好不喜欢花哨、不喜欢社交元素、不要游戏里那种"假努力"的鸡汤
对 AI 的态度拥抱 AI，但警惕 AI 编数据（幻觉），需要"看得见的确认"

核心价值主张

🎯 One-Liner

"让记账这件事不再消耗意志力，让每一笔账都有出处，让信用卡不再逾期，让攒钱变成一场可量化的游戏。"

A+ 级产品的定义（自我标准）

一个 A+ 级记账 App 必须同时满足：

  • 快 — 3 秒内完成一笔记账（手动或 AI 识别）
  • 准 — 数据 100% 可信，账目零错误
  • 懂 — 不只是记账，而是给财务洞察
  • 黏 — 每天主动想打开，不是被逼打开
  • 美 — 视觉精致但不浮夸，专业感强
💡 与咔皮记账的差异化

咔皮记账是「通用记账工具」，我们要做的是「私人财务操作系统」。区别：

  • 咔皮：记账 + 简单统计 → 我们：记账 + AI 审计 + RPG 激励 + 深度分析
  • 咔皮：手动为主 → 我们：AI 自动识别 + 人工二次确认（兼顾效率和准确）
  • 咔皮：无信用卡专项 → 我们：信用卡账期、还款日、账单日、可用额度全维度管理

咔皮已知缺陷（我们要解决的痛点）

缺陷用户反馈我们的解决方案
导航层级不直观 新用户上手需适应 底部 5 个 Tab 极简结构，符合 iOS 标准
账单修改不便 只能改不能加减 支持"修改金额为退款"快捷操作，自动生成负数账目

信用卡还款无总结 用户明确反馈的痛点 信用卡页是核心 Tab 之一，账期日历 + 还款历史 + 信用报告
手动记账流程不顺 主推自动记账但手动不顺 手动记账是 P0 重点，≤ 5 秒、3 步完成
未接入理财平台 数据孤岛 暂不接入（自用，不依赖第三方 API）
功能相对较少 成熟记账功能未补齐 预算 + 信用卡 + RPG + AI 审计全覆盖

功能策略（开发优先级依据）

✅ 三条铁律
  1. 咔皮有的好功能必须有：用户已经习惯这些功能，不能缺（语音/拍照/OCR/多账本）
  2. 咔皮的痛点必须解决：导航乱、修改不便、不支持转账、无信用卡深度——这些必须做到更好
  3. 咔皮没有的优势要加：RPG 化、审计官、纯本地隐私、0 成本长期可用

对咔皮用户的迁移友好性

虽然我们不发布到 App Store（侧载），但要确保从咔皮导出的数据能无缝导入：

  • 支持咔皮 CSV 格式导入（金额/分类/日期/账户映射）
  • 分类映射：咔皮的"餐饮"→ 我们的"餐饮"
  • 账户映射：咔皮的"现金"→ 我们的"现金"
  • 导入后用户可手动调整差异
⚠️ 与咔皮的差异化不能为差异化而差异化

我们的所有差异化都必须服务于"用户痛点"，不是为了不同而不同：

  • RPG 化 → 解决"记账坚持不下来"
  • 审计官 → 解决"AI 幻觉不放心"
  • 本地化 → 解决"隐私担忧"
  • 0 成本 → 解决"怕被停服"
  • 重度信用卡管理 → 解决"还款总忘、统计混乱"

03AI 身份系统设计

身份架构：双层结构

采用「底层审计官 + 外壳攒钱合伙人」双层架构，审计官永远是真相之源，攒钱合伙人是情感接口。

🛡️ 底层：审计官（"真相之源"）

无性格、无表情、只输出事实。所有数据校验、异常检测、账目核对都走它。用户看到的"警告"、"异常"全部由它发出。

🎮 外壳：攒钱合伙人（"情感接口"）

重度 RPG 化的虚拟角色。所有主动推送、鼓励、调侃、剧情都走它。但它说的每一句话都必须基于审计官的真相。

角色人格设定

审计官人格

姓名：审计官
代号：AUDIT-01
性格：极度严谨、不苟言笑、用数据说话、对错误零容忍
语言风格：陈述句、简短、不带感情色彩

对话原则：
  ✅ "餐饮类支出本月累计 1847 元，超预算 23.4%。"
  ✅ "检测到异常：2026-07-13 23:47 单笔支出 ¥899，超过日均支出 18 倍。"
  ❌ "哎呀，你今天花太多了哦~"
  ❌ "别担心啦，还剩一点点预算嘛"

触发场景：
  - AI 自动识别账目 → 让用户确认
  - 异常消费 → 红色警告
  - 信用卡账单日/还款日前 → 精确倒计时
  - 每月 1 号 → 出具"财务体检报告"

攒钱合伙人人格（重度 RPG 化）

姓名：可自定义（默认"攒攒"）
形态：Q 版像素角色，会随等级成长换形态
初始状态：一级"小财苗"（绿色幼苗 + 单片叶子）
满级状态：十级"财神"（穿着金甲的元宝形象）

人格双面：
  - 70% 时间：温柔鼓励型（"主人好棒！连续 7 天未超预算~"）
  - 30% 时间：毒舌调侃型（"主人又在喝奶茶？这是本月的第 6 杯。"）

台词库（节选）：
  🎉 成就解锁：
     "🎊 解锁【预算执行者】！本月光荣结余 ¥234！"
     "👑 解锁【10 级财神】！你已经超越 99% 的记账用户。"

  💸 预算警告：
     "⚠️ 餐饮预算还剩 ¥89，还能吃 3 顿工作日午餐哦~"
     "🚨 主人！本月已花 95%！建议启动'吃土模式'。"

  💳 还款提醒：
     "📅 信用卡 X 还剩 3 天还款，金额 ¥2350，已准备好。"
     "✅ 还款完成！信用值 +50，主人信用等级：A+"

  🏆 剧情推进：
     "📖 主线任务：攒下第一笔 ¥10000 → 进度 73.4%"
     "✨ 支线解锁：【咖啡自由挑战】30 天不点外卖咖啡"

禁止：
  ❌ 编造数据
  ❌ 给虚假鼓励（预算超了还说"没关系"）
  ❌ 鸡汤话术（"钱不是问题，爱才是"）

04核心功能矩阵

按 P0 (必须有) · P1 (应该有) · P2 (最好有) 三级优先级划分。MVP（最小可用版本）只做 P0。

P0 - MVP 必须有（v1.0 上线）

功能说明验收标准
P0 手动记账最基础的记账，3 秒内完成点击 → 选分类 → 输入金额 → 保存 ≤ 3 步
P0 分类管理预置 + 自定义分类，支持图标至少 12 个预置分类，支持增删改
P0 信用卡管理添加多张信用卡，记录账单日/还款日/额度支持 ≥ 3 张卡，账期自动计算
P0 还款提醒还款日前 3 天 + 当天推送通知本地通知 + 应用内红点双通道
P0 预算管理月度总预算 + 分类预算双层支持设置 → 执行 → 超支警告
P0 基础统计本月汇总、分类占比、近 30 天趋势3 种图表（饼图、柱状图、折线图）
P0 数据本地存储SQLite 加密存储SQLCipher 加密，App 卸载前数据不丢
P0 数据导出CSV 导出，备份到 iCloud/Files一键导出，文件可被 Numbers/Excel 打开

P1 - 应该有的（v1.1 迭代）

功能说明
P1 OCR 自动记账截图识别微信/支付宝账单，用户确认后入库
P1 多账本日常/旅行/生意分开，独立统计
P1 转账记录账户间转账不重复计入收支
P1 资产账户现金、银行卡、余额宝等账户管理
P1 月度财务报告每月 1 号生成上月报告，可导出 PDF
P1 RPG 等级系统经验值、等级、解锁成就
P1 信用卡账期视图可视化账期日历，本期已花/可用额度

P2 - 锦上添花（v2.0+ 远期）

功能说明
P2 AI 财务洞察消费模式识别、异常预警、攒钱建议
P2 跨账本分析"旅行账本超支 30%，但日常账本盈余"
P2 主线剧情任务"攒下第一笔 ¥10000" 等长线目标
P2 角色皮肤系统攒攒换装，但不影响功能
P2 Apple Watch 组件手表端快速记一笔
P2 iCloud 同步多设备同步（自用基本用不到）

05用户旅程与核心流程

首次启动 Onboarding

欢迎页 → 选择角色形态 → 设置总预算 → 添加信用卡 → 录入资产 → 首笔记账教学 → 主页

核心场景：日常记账

场景 A：手动记账（最常用）

主页点击 + → 选分类（图标网格） → 输入金额（数字键盘） → 可选备注+支付方式 → 保存（震动反馈）

目标：≤ 5 秒完成

场景 B：截图自动识别（v1.1）

截图保存到相册 → App 内"扫一扫" → OCR 提取信息 → 审计官校验 → 用户确认/修改 → 入库

关键：每步都可回退，必须用户确认

核心场景：信用卡还款

06系统架构设计

技术选型

层技术选型理由
前端框架Flutter 3.x一套代码，Windows 上写，macOS 编译，未来可上 Android
状态管理Riverpod 2.x比 Provider 更现代，类型安全，测试友好
本地数据库Drift (SQLite) + SQLCipherDrift 是 Flutter 最成熟的 ORM，SQLCipher 加密
本地通知flutter_local_notificationsiOS 16+ 完美支持
图表fl_chartFlutter 生态最专业的图表库
OCR（v1.1）Apple Vision API（原生 iOS）+ MethodChannel免费、本地、隐私好，准确率高
路径规划go_router声明式路由，深度链接支持
国际化flutter_localizations预留英文支持
💡 为什么不用 React Native / 原生 Swift
  • vs 原生 Swift：开发速度慢，且不能在 Windows 上写代码
  • vs React Native：Flutter 性能更稳定，UI 一致性更好，RPG 化需要复杂动画
  • vs Tauri / 其他：移动端体验 Flutter 完胜

⚠️ 关键技术决策

OCR 用本地 Vision API 而非云端 LLM：

  • 成本：Vision API 完全免费，调用云端 LLM 要花钱，违背 0 成本原则
  • 隐私：账单截图是敏感数据，本地处理更安全
  • 准确率：账单格式相对固定，Vision API 对印刷体识别率 > 95%
  • LLM 仅用于"非结构化文本理解"（如 OCR 后二次校验）

07UI/UX 设计规范

设计原则

1. 信息密度优先

每屏信息量足够，但不拥挤。专业感来自"看得清"，不是"看得花"

2. 操作永远 ≤ 3 步

核心操作不超过 3 次点击。任何超过 3 步的操作都是设计失败

3. 反馈必须即时

每个动作都有视觉/触觉反馈。无反馈 = 用户不知道发生了什么

4. RPG 是"皮肤"

所有 RPG 元素都不能阻挡效率。游戏化是奖励，不是负担

5. 暗色为主

默认暗色主题（专业感 + 省电），可切换亮色

6. 数据可视化大于文字

能画图就不写文字，能用数字就不用形容词

CREATE TABLE account (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,           -- "招商信用卡" "现金" "余额宝"
  type TEXT NOT NULL,           -- 'cash' | 'debit' | 'credit' | 'investment'
  balance REAL NOT NULL DEFAULT 0,  -- 当前余额（信用卡为可用额度）
  icon TEXT,                    -- 图标 emoji
  color TEXT,                   -- 主题色
  credit_limit REAL,            -- 信用卡总额度
  statement_day INTEGER,        -- 账单日（信用卡专用，1-31）
  due_day INTEGER,              -- 还款日（信用卡专用，1-31）
  is_archived INTEGER DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL
);

CREATE INDEX idx_account_type ON account(type);

category（分类）

CREATE TABLE category (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,           -- "餐饮" "交通" "工资"
  type TEXT NOT NULL,           -- 'expense' | 'income' | 'transfer'
  icon TEXT,                    -- emoji
  color TEXT,
  parent_id INTEGER,            -- 支持二级分类（餐饮 → 早餐/午餐/晚餐）
  is_system INTEGER DEFAULT 0,  -- 系统预置 vs 用户自定义
  sort_order INTEGER DEFAULT 0,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  FOREIGN KEY (parent_id) REFERENCES category(id)
);

transaction（交易）

CREATE TABLE transaction (
  id INTEGER PRIMARY KEY,
  account_id INTEGER NOT NULL,
  category_id INTEGER NOT NULL,
  amount REAL NOT NULL,         -- 正数表示收入，负数表示支出
  type TEXT NOT NULL,           -- 'expense' | 'income' | 'transfer'
  note TEXT,
  occurred_at INTEGER NOT NULL, -- 交易发生时间
  transfer_pair_id INTEGER,     -- 转账配对 ID
  source TEXT,                  -- 'manual' | 'ocr' | 'siri' | 'shortcut'
  confidence REAL,              -- OCR 识别置信度 0-1
  is_confirmed INTEGER DEFAULT 1,  -- OCR 账目需用户确认
  image_path TEXT,              -- 关联截图（OCR 时）
  location_lat REAL,            -- 可选位置
  location_lng REAL,
  created_at INTEGER NOT NULL,
  updated_at INTEGER NOT NULL,
  FOREIGN KEY (account_id) REFERENCES account(id),
  FOREIGN KEY (category_id) REFERENCES category(id)
);

OCR 自动记账流程

🔍 流程：截图 → 入库
  1. 截图保存：用户截图微信/支付宝账单
  2. App 内唤起 OCR：用户点击"扫一扫"或相册选择
  3. Vision API 识别：提取金额、商户、时间
  4. 审计官校验：
    • 金额合理性（不是 0、不是天文数字）
    • 商户名是否在已知库中
    • 时间是否在合理范围（不是未来）
    • 是否与近期交易重复
  5. 用户确认界面：显示识别结果，用户可修改
  6. 攒攒说话："已识别餐饮 ¥58，是否确认？"
  7. 入库：用户点确认后写入数据库

异常检测规则（审计官核心）

class AuditRule {
  // 1. 金额异常检测
  bool checkAmountAnomaly(Transaction tx) {
    final avgDaily = getAvgDailyExpense(tx.accountId);
    if (tx.amount.abs() > avgDaily * 5) {
      return true;  // 单笔超过日均 5 倍，触发红色警告
    }
    return false;
  }

  // 2. 时间异常检测
  bool checkTimeAnomaly(Transaction tx) {
    final hour = DateTime.fromMillisecondsSinceEpoch(tx.occurredAt).hour;
    if (hour >= 1 && hour <= 5 && tx.category == '餐饮') {
      return true;  // 凌晨 1-5 点餐饮，触发黄色提示
    }
    return false;
  }

  // 3. 预算超支检测
  bool checkBudgetAnomaly(Transaction tx) {
    final usage = getBudgetUsage(tx.categoryId);
    if (usage >= 0.95) {
      return true;  // 预算用 95%，触发橙色警告
    }
    return false;
  }

  // 4. 重复检测
  bool checkDuplicate(Transaction tx) {
    final recent = getRecentTransactions(
      accountId: tx.accountId,
      categoryId: tx.categoryId,
      amount: tx.amount,
      within: Duration(hours: 1),
    );
    return recent.isNotEmpty;  // 1 小时内同金额同分类
  }

  // 5. 信用卡账期检测
  bool checkCreditLimit(Transaction tx) {
    if (getAccountType(tx.accountId) == 'credit') {
      final usage = getCreditUsage(tx.accountId);
      if (usage >= 0.9) {
        return true;  // 信用卡额度用 90%
      }
    }
    return false;
  }
}
